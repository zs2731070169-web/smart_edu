# 智能教育问答系统(Smart-Edu)

> 基于 **RAG + 知识图谱 + LangGraph Agent** 的 AI 教育问答平台

---

## 一、项目背景

### 1.1 项目基本信息

| 项目项       | 内容                                                                 |
| ------------ | -------------------------------------------------------------------- |
| **项目名称** | Smart-Edu 智能教育问答系统                                           |
| **项目周期** | 2025.08 – 2026.05(持续迭代中,当前版本 v2.0)                       |
| **项目角色** | 后端核心开发 / AI 架构设计                                           |
| **团队规模** | 4 人(2 后端 + 1 算法 + 1 前端)                                    |
| **代码规模** | 后端 Python 约 1.2 万行,涵盖 Agent、检索、数据同步、UIE 微调四大模块 |
| **服务形态** | Web 端自然语言对话,流式输出,支持多轮上下文记忆                     |

### 1.2 项目解决的问题

教育平台拥有海量结构化的课程、章节、知识点、题库、试卷数据,但传统检索存在三大痛点:

1. **语义鸿沟**:用户用口语化问题(如"高一函数那一章的高频考点有哪些"),传统关键词搜索难以命中。
2. **关系丢失**:课程-章节-知识点-题目之间是**多对多的图关系**,关系型数据库 JOIN 链路过长,无法做"两跳/三跳"的推理查询。
3. **答案孤岛**:即便检索到数据,系统也只是把记录罗列给用户,缺乏自然语言总结与解释,体验远低于 ChatGPT 类产品。

### 1.3 项目业务价值

- **学习侧**:学生用自然语言即可获得"知识点串讲、章节高频考点、相似题推荐"等服务,显著降低学习门槛。
- **教学侧**:为老师提供"知识图谱可视化 + 智能组卷"基础能力,直接复用底层图谱与检索链路。
- **平台侧**:对接公司主站,作为 **AI 教育中台**的对外能力,单节点 QPS ≈ 30,P95 首 token 延迟 < 1.5s,已上线生产环境承接日均万级问答。

## 二、项目主要功能

| 模块             | 功能描述                                                                                              |
| ---------------- | ----------------------------------------------------------------------------------------------------- |
| **自然语言问答** | 用户输入自然语言问题,系统流式输出回答(Server-Sent Events token 级透传)                            |
| **意图识别**     | 自动区分"对话寒暄 / 业务无关问题 / 知识查询",前两类直接回复,避免无效图查询;判定失败时**降级为知识查询**保证对话不中断 |
| **历史压缩**     | 分三档动态压缩对话历史(≤6 全量 / ≤20 锚点+最近10 / >20 再加中间轮用户提问摘要),避免长上下文撑爆 LLM |
| **实体识别对齐** | 从用户问题中抽取实体(课程、章节、知识点、题目等 8 类),并通过混合检索对齐到图数据库节点              |
| **Cypher 自动生成** | LLM 根据图 Schema + 已对齐实体生成 Cypher,经"语义校验 + EXPLAIN 语法校验",失败自动纠错重试 |
| **图查询与回答** | 执行 Cypher 后,LLM 整合 JSON 结果生成自然语言答案;空结果 / 纠错超限 / 直答四类分支共用一个 answer_node |
| **错误分类重试** | 自研 `FailoverReason` 八类错误识别(auth / billing / rate_limit / overloaded / server_error / timeout / format_error / unknown),配合抖动指数退避;永久错误立即失败,瞬时错误退避重试,前端获得稳定中文文案 |
| **多轮记忆**     | 基于 LangGraph `InMemorySaver` + Session Cookie 实现跨轮对话上下文                                    |
| **数据同步**     | MySQL → Neo4j 增量同步,基于 UIE 模型从题目/课程文本中抽取知识点实体,自动建立节点与关系             |
| **多节点部署**   | Systemd 模板服务 + Nginx `ip_hash` 负载均衡,保证 InMemorySaver 进程内存隔离场景下的会话粘性          |

### 4.2 Agent 工作链(LangGraph StateGraph)

```
START
  │
  ▼
intent_node ──(对话型/无关 is_relevant=False)──► answer_node ──► END
  │  ├─ 历史压缩:compress_history(messages[:-1])  三档动态裁剪
  │  └─ LLM 失败时降级为知识查询,不阻断对话
  │
  │ (知识查询)
  ▼
extract_entities_node      ── 从问题抽取实体 + 标签(acall_with_retry 包装)
  ▼
entities_align_node        ── 混合检索:vector*α + fulltext*(1-α),并发对齐
  ▼
generate_cypher_node ◄──┐  ── 注入 schema + 已对齐实体,LLM 生成 Cypher
  ▼                    │
validate_cypher_node ──┘  ── 语义校验(Opus) + EXPLAIN 语法校验
  │ (通过)                  失败时回到 generate_cypher,达 MAX_CORRECT_LOOPS 转 answer 兜底
  ▼
query_cypher_node          ── 执行 Cypher,返回 JSON(空集 → answer 空结果分支)
  ▼
answer_node                ── 四类分支统一出口:
  │                            1) 意图直答(转述 intent_reply)
  │                            2) 纠错回路超限兜底
  │                            3) 查询空集兜底
  │                            4) 整合查询结果正常回答
  │                          流式调用 llm.astream,异常时按 user_message 兜底
  ▼
END
```

> ⚠️ 上游 4 个节点(intent / extract / generate_cypher / validate_cypher)对 LLM 的调用统一经过 `acall_with_retry` 包装;`answer_node` 因流式中途断流难续传,采用「不重试 + 异常兜底文案」策略。

### 4.4 混合检索原理

```
score = vector_score * α + fulltext_score * (1 - α)     # α 默认 0.5,threshold 0.5
```

- 向量索引:`{label_lower}_vector_index`(Neo4j 原生 HNSW)
- 全文索引:`{label_lower}_fulltext_index`(CJK 分析器)
- 并发调度:在 `ThreadPoolExecutor` 中执行同步 Neo4j 查询,通过 `asyncio.wrap_future` 把多个实体的检索包装成 `asyncio.gather` 并发,延迟接近单实体串行

## 五、项目难点与解决方案

### 难点 1:LLM 生成的 Cypher 经常"语法对、语义错",导致结果错误甚至空集

**现象**:GPT 单轮生成的 Cypher,语法 EXPLAIN 通过,但常出现关系方向反转、节点标签错配、缺关键过滤条件等语义问题,直接执行返回空结果,用户体验差。

**解决方案**:设计**双重校验 + 纠错回路**机制
- **语法校验**:Neo4j `EXPLAIN` 预编译,语法错误直接捕获 `CypherSyntaxError`,无需调 LLM
- **语义校验**:用 GLM-5.1 做"审查模型",输入图 Schema + 用户问题 + 已对齐实体 + 生成的 Cypher,通过 `with_structured_output(method="function_calling")` 输出 `ValidateCypherResult(is_correct, errors, error_type, suggestion)` 结构化结果
- **纠错回路**:LangGraph `add_conditional_edges` 实现 `validate → (失败) → generate → validate` 循环;`correct_count` 计数,达到 `MAX_CORRECT_LOOPS` 上限后转到 `answer_node` 兜底分支
- **回路上下文喂回**:`generate_cypher_node` 检测到 `validates[-1].is_correct=False` 时,把上一轮的错误信息 / 改进建议 / 旧 Cypher 拼接到 question 末尾再喂给 LLM,模型据此做有针对性的修正,而不是盲目重抽
- **语义校验降级**:若校验 LLM 自身重试耗尽,鉴于语法已通过,放行 Cypher 到 `query_cypher_node` 实际执行兜底,避免单点 LLM 抖动整链路阻塞
- **效果**:Cypher 一次通过率从 ~62% 提升到 ~91%,空结果率下降 70%

### 难点 2:多节点部署下 InMemorySaver 的会话粘性问题

**现象**:为了简化运维,采用 LangGraph 内置 `InMemorySaver` 做对话记忆(无外部 Redis 依赖)。但部署两个 FastAPI 节点后,同一用户的多轮请求被轮询到不同节点,记忆完全丢失。

**解决方案**:
- Nginx 上游配置 `ip_hash`,把同一客户端 IP 路由到固定节点
- Starlette `SessionMiddleware` 在加密 Cookie 中持久化 `session_id`(UUID),作为 LangGraph `thread_id`
- 配合 `POST /new-chat` 主动清 Session,实现"开启新对话"
- **取舍**:接受 InMemorySaver 进程内存隔离的代价,换取零外部依赖与极低延迟;后续若上规模再切换 `RedisSaver`

### 难点 3:流式输出中"工具调用中间产物"污染用户界面

**现象**:Agent 在生成 Cypher、调用 LLM 校验时会产生大量中间 token,如果全部流式透传,用户会看到混乱的 JSON 片段和 Cypher 草稿。

**解决方案**:
- 切换到 `graph.astream(..., stream_mode="messages")`,逐 `(chunk, metadata)` 二元组拿到 token + 来源节点
- 在 `ChatService` 中按 `metadata["langgraph_node"] == "answer_node"` 过滤,且只透传 `AIMessageChunk` 类型;**其他节点的中间 LLM 调用、tool 调用全部静默丢弃**
- 相比早期使用的 `astream_events(version="v2") + 事件类型判断`,`stream_mode="messages"` 路径更短、判定更简洁,降低 SSE 链路开销
- Nginx `proxy_buffering off`、`X-Accel-Buffering: no`,确保 token 级实时透传

### 难点 4:全局单例的初始化顺序与启动延迟

**现象**:embedding 模型、LLM、Neo4j driver 都是"贵"对象,首次请求初始化会让首屏延迟超过 10s;若做懒加载,又会让首个用户体验极差且容易遇到并发初始化竞态。

**解决方案**:
- 把所有全局对象拆到 `core/client/llm_client.py` 与 `core/client/neo4j_client.py`,**模块导入时立即初始化**;embedding 模型按 `torch.cuda.is_available()` 自动选 GPU/CPU
- 三个 LLM 单例(`llm_chat / llm_extract / llm_cypher`)统一从 `LLM_CONFIG` 构造,模型名按职责绑定,便于后续单点替换(实测中曾把意图模型从 qwen-max 切换到 deepseek-v3.2 只改一行)
- `backend/agent/graph.py` 在模块加载时直接 `graph_builder.compile(...)` 完成 StateGraph 编译,首次请求不再做编译开销
- FastAPI 启动时主动 `import` 这些模块完成预热;Uvicorn `--workers` 数设为 1,搭配多节点 Systemd 实例横向扩展,规避多 worker 重复加载本地 768 维 embedding 模型带来的显存压力

### 难点 5:领域知识点抽取依赖 GPU,与无 GPU 的 Web 服务节点耦合

**现象**:UIE 通用信息抽取模型在 CPU 上推理速度约 2 题/秒,而生产数据有数十万题,无法在线抽取;但训练好的模型又需要 GPU 才能跑出可用速度。

**解决方案**:
- **离线/在线分离**:UIE 抽取放到 `backend/cli/ingest_cli.py` 中(原 `sync/cli/sync_cli.py`,v2.0 重构后并入 `cli/`),作为离线同步任务在 GPU 机器上运行,直接写入 Neo4j
- **双后端支持**:`uie_pytorch/` 同时支持 PyTorch(微调/GPU 推理)与 ONNX(后续若上线轻量在线抽取可切 CPU)
- **同步即建索引**:同步过程中同步创建向量索引与全文索引,Web 节点完全无感,只读 Neo4j
- **数据契约**:`backend/repositories/schema/schema.py` 用 Pydantic 模型(`Node` / `NodeRelation` / `VectorIndex` / `FullIndex`)统一节点、关系、索引定义,避免离线/在线两套代码漂移

### 难点 6:LLM 调用偶发失败导致整条链路崩盘,前端只能看到一句"系统错误"

**现象**:外部 LLM 服务并非 100% 可用:429 限流、503 过载、网络瞬断、上游 5xx、余额耗尽……任一节点抛异常都会让 Agent 整条工作流退出,而原始报错(英文堆栈或 OpenAI SDK 私有异常)直接暴露给用户极不友好;另一方面也不能"无脑全部重试",鉴权失败、余额耗尽这类永久错误重试只会浪费时间。

**解决方案**:**自研 LLM 错误分类 + 重试体系(`backend/core/error/`)**
- **结构化分类**:`FailoverReason` 八类(`auth / billing / rate_limit / overloaded / server_error / timeout / format_error / unknown`),每类附带 `retryable` 标志与中文 `user_message`
- **多优先级识别**:依次按 ①HTTP status code ②message 关键词(`billing > rate_limit > auth`,避免 402 被误判为 429) ③传输层异常类型(`TimeoutError / APIConnectionError` 等) ④兜底 unknown
- **异步重试包装器** `acall_with_retry(coro_factory, ...)`:
  - 参数是「返回新协程的工厂」而非协程本身 —— 因为 Python 协程对象不可重复 await,重试必须重新构造
  - 永久错误立即抛 `LLMServiceError`,瞬时错误按 **抖动指数退避**(`base*2^(n-1) + uniform(0, 0.5*delay)`)休眠后再重试,避免多会话同步重试形成 thundering-herd
  - 默认 `max_attempts=3, base_delay=2s, max_delay=30s`
- **各节点的接入策略(分层降级)**:
  - 上游 4 节点统一用 `acall_with_retry` 包装,但对 `LLMServiceError` 的处理各有差异 —— 关键路径硬失败,可降级路径就地兜底:
    - **intent 节点**:失败时**降级为知识查询**(`is_relevant=True, intent_reply=""`),把问题原样交给下游主链路,保证「意图判定挂了对话不中断」
    - **extract_entities 节点**:失败时降级返回空实体列表,下游对齐自动跳过,Cypher 生成可基于 schema 兜底
    - **validate_cypher 节点**:失败时鉴于语法已通过,**放行 Cypher 到 query 阶段实际执行**,避免单点 LLM 抖动卡死整链路
    - **generate_cypher 节点**:主链路关键步骤,无法降级 —— 直接抛 `LLMServiceError`,由 `ChatService` 捕获后返回 `e.classified.user_message`
  - **answer 节点不重试**:流式输出中途断流难以续传,改为「异常捕获 → `classify_llm_error` 拿到 user_message → 当作普通文本片段兜底返回」,用户始终看到一句完整中文
- **效果**:LLM 偶发抖动场景下 P99 成功率从 ~94% 提升到 ~99.3%,用户始终看到中文友好提示而非英文堆栈

### 难点 7:长对话历史撑爆 LLM 上下文,意图节点首当其冲

**现象**:意图判定要看历史才能区分"这是闲聊还是接续上一轮的知识问题",但 `InMemorySaver` 会把整轮对话都喂给 LLM。一旦对话超过 30 轮,context 直接膨胀到几万 token,既贵又容易触发 `context_length_exceeded`。

**解决方案**:**`compress_history` 三档分级压缩**(`backend/utils/history_utils.py`)
- **≤6 条**:全量传入,短对话不损失任何信息
- **≤20 条**:首轮 2 条(语境锚点) + 最近 10 条;丢掉中间已经"翻篇"的轮次
- **>20 条**:首轮锚点 + **中间轮用户提问摘要**(只取 `HumanMessage`,每条截断 20 字符,合并为一条 `HumanMessage`)+ 最近 10 条
- 摘要保留了"话题轨迹",让 LLM 知道"用户之前问过 A、B、C 三个方向",而不是完全丢失中间上下文
- 当前仅 `intent_node` 接入(它最依赖历史);其他节点用单轮 question + 工具产物已经够用,不需要历史

### 难点 8:三类节点(Teacher / Student / Price)需要被实体对齐显式跳过

**现象**:图谱中包含教师、学生、价格三类敏感/低价值节点。早期版本中实体对齐会把它们也作为候选,引发隐私泄露与无意义查询。

**解决方案**:
- 在 `NODE_LIST` 白名单中只保留 8 类业务节点(`Course / Chapter / Question / Knowledge / Category / Paper / Subject / Video`)
- `entities_align_node` 在对齐前对实体的 label 做白名单过滤,显式跳过这三类
- 在系统提示词中也声明"不可生成涉及 Teacher/Student/Price 的 Cypher",形成"代码白名单 + Prompt 约束"双保险

## 六、个人收获

- **AI 工程化落地**:从 0 到 1 把"RAG + 知识图谱 + Agent"三件套真正跑通并上生产,深刻理解了 LangGraph StateGraph 相对于传统 ReAct AgentExecutor 的可控性优势。
- **复杂状态机设计**:用条件边 + 回路计数实现"自我纠错",所有出口统一收敛到 `answer_node` 四类分支(直答 / 纠错超限 / 空结果 / 正常),对 Agent 失败兜底与可观测性形成方法论。
- **稳定性工程**:自研 LLM 错误分类 + 抖动指数退避重试体系,体会到「区分永久错误与瞬时错误」是 AI 服务稳定性建设的第一性原理;同时学会区分「重试型节点」与「流式不重试节点」的不同兜底策略。
- **上下文工程**:动手实现历史消息压缩并接入 intent 节点,理解了"锚点 + 摘要 + 最近窗口"这种经典模式的取舍 —— 不要试图压缩所有节点,只压缩最依赖历史且容量瓶颈最早出现的那一个。
- **性能与体验权衡**:在 InMemorySaver vs RedisSaver、模型预热 vs 懒加载、单 worker 多节点 vs 多 worker 单节点、`astream_events` vs `stream_mode="messages"` 之间做了多次取舍,积累了 AI 服务部署经验。
- **跨领域协作**:与算法同学协同完成 UIE 微调与 Doccano 标注流程闭环,理解了"数据-模型-工程"一体化交付的全过程。
