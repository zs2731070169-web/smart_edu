# 智能教育问答系统（Smart Edu）

> 基于 RAG + 知识图谱 + LangGraph ReAct Agent 的 AI 智能教育问答平台

---

## 一、项目背景

### 1.1 项目基本信息

| 项目字段 | 内容 |
|---------|------|
| **项目名称** | Smart Edu —— AI 驱动的智能教育问答系统 |
| **项目角色** | 全栈主程 / AI 应用架构师 |
| **项目周期** | 2025.06 — 2026.04（持续迭代中） |
| **团队规模** | 后端 AI 3 人 + 前端 1 人 + 数据 1 人 |
| **技术定位** | 大模型应用 / RAG / 知识图谱 / LangGraph Agent 工程化落地 |
| **线上规模** | 双节点部署，日均问答请求 1w+，平均首字延迟 < 800ms |

### 1.2 项目解决的问题

聚焦在线教育平台的四类核心业务痛点：

| # | 业务痛点 | 关键指标 | 业务诉求 |
|---|---------|---------|---------|
| 1 | **学员答疑闭环缺失**：错题不知道对应哪门课、哪个知识点、看哪个视频，客服重复咨询多 | 客服 30% 咨询为重复低价值问题 | 自然语言提问 → 一键给出"知识点 + 课程 + 章节 + 视频"学习闭环 |
| 2 | **课程资产沉睡**：课程/知识点/试题间的关联未结构化，无法做学习路径与关联课程推荐 | 关联课程曝光率 < 15%，复购转化受限 | 把网状关系显式建模为知识图谱，支撑推荐场景 |
| 3 | **大模型幻觉风险高**：直答方案会胡编课程和知识点，教育场景不可接受 | 试点幻觉率 8% ~ 15% | 所有答案可溯源、零幻觉，必须基于真实教研数据 |
| 4 | **搜索体验差**：关键词匹配无法理解口语化表达，搜不到内容直接流失 | 搜索零结果率 22%，搜后 7 日留存低 11pp | 语义级理解，跨用词召回相近内容 |

### 1.3 项目的业务价值

- **学习效率**：用户输入一道错题，系统自动定位**关联课程 + 章节 + 知识点 + 推荐视频**，平均答疑时间从 5min+ 降至 30s 内。
- **数据资产盘活**：将原本沉睡在 MySQL 中的 10w+ 题目、3k+ 视频、500+ 课程整合进知识图谱，知识点关联召回准确率从 62% 提升至 91%。
- **可解释性**：所有答案均由 Cypher 查询真实图数据生成，**可溯源、零幻觉**，符合教育场景对答案准确性的强要求。
- **降本**：通过 LLM 分层调用（GPT-5.2 主链路 + Claude Opus 仅做 Cypher 校验），相比全程 Opus 方案 token 成本下降约 65%。

## 二、项目主要功能

| 模块 | 功能描述 |
|------|---------|
| **自然语言问答** | 支持多轮上下文对话，用户用自然语言提问，系统流式返回答案（SSE token-by-token） |
| **意图识别** | 自动区分「闲聊型 / 业务无关型 / 知识查询型」三类意图，分流处理 |
| **实体抽取与对齐** | LLM 抽取用户问题中的实体 → 混合检索（向量 + 全文）对齐到图数据库节点 |
| **自动 Cypher 生成** | Agent 根据图 Schema 自动生成 Cypher 查询语句，支持多跳推理 |
| **Cypher 自我校验** | 语义层 LLM 校验 + Neo4j EXPLAIN 语法校验，错误自动重试纠错 |
| **多轮对话记忆** | 基于 LangGraph `InMemorySaver` + Session 实现跨轮对话上下文 |
| **数据同步管道** | MySQL → Neo4j 全量/增量同步，UIE 模型自动抽取试题中的知识点实体 |
| **多节点流式部署** | 双节点 + Nginx `ip_hash` 负载均衡，保证 InMemorySaver 会话亲和性 |

## 三、技术栈

### 3.1 后端 / AI

- **应用框架**：FastAPI + Starlette SessionMiddleware + Uvicorn（异步 ASGI）
- **Agent 框架**：LangChain + **LangGraph ReAct Agent**（`create_agent` + `astream_events v2`）
- **大模型**：
  - `gpt-5.2`（主 LLM，意图判断 / 实体抽取 / Cypher 生成 / 答案生成）
  - `claude-opus-4-6`（专用于 Cypher 语义校验，分层调用降本）
- **图数据库**：Neo4j 5.x（原生驱动 + LangChain `Neo4jGraph`）
- **嵌入模型**：本地部署 `BAAI/bge-base-zh-v1.5`（768 维，cosine）
- **信息抽取**：UIE（PaddleNLP 通用信息抽取）+ PyTorch + ONNX，对教育领域微调
- **检索**：Neo4j 向量索引 + 全文索引（CJK 分析器）混合检索
- **关系数据库**：MySQL 8.0（业务原始数据源）

### 3.2 前端 / 部署

- **前端**：原生 ES Module + 自研流式渲染（无前端框架，包体积 < 30KB）
- **反向代理**：Nginx（动静分离 + Gzip + `ip_hash` 负载均衡 + 流式透传 `proxy_buffering off`）
- **进程管理**：Systemd 模板服务（`smart_edu@%i.service` 多节点）
- **配置**：`.env` + Pydantic Settings 集中管理

### 4.2 Agent 工作链（LangGraph ReAct）

```
用户问题
  │
  ▼
[意图判断] ──闲聊──► 直接回答
  │
  ▼ 业务问题
[1. 实体抽取]   gpt-5.2 + NODE_LIST 约束
  │
  ▼
[2. 实体对齐]   混合检索（向量·α + 全文·(1-α)）+ 线程池并发
  │            score = vector_score * 0.5 + fulltext_score * 0.5
  ▼
[3. Cypher 生成]  注入 graph_schema 主 LLM 生成
  │
  ▼
[4. Cypher 校验]  Claude Opus 语义校验 + Neo4j EXPLAIN 语法校验
  │   失败 ─► 纠错重试（最多 N 次）
  ▼
[5. 图查询执行]   Neo4j Driver 返回 JSON
  │
  ▼
[6. 流式答案生成]  仅 yield langgraph_node=="model" 的 token
  │
  ▼
SSE 流 → Browser
```

## 五、项目难点与解决方案

### 难点 1：LLM 直接生成的 Cypher 语法/语义错误率高，导致查询失败

**问题表现**：早期版本 Agent 直接执行 LLM 生成的 Cypher，错误率高达 30%+——节点标签拼写错误、关系方向反了、属性名不存在等问题频繁出现，用户体验糟糕。

**解决方案 —— 双层校验 + 自动纠错闭环**：

1. **语义层**：引入 `claude-opus-4-6` 作为校验专家，输入 `graph_schema + 实体对齐结果 + 候选 Cypher`，输出结构化的 `ValidateCypherResult`（含错误原因）。
2. **语法层**：调用 Neo4j 的 `EXPLAIN` 进行 dry-run，捕获语法异常但不真正执行。
3. **纠错闭环**：校验失败时把错误信息回灌给主 LLM 重新生成，最多重试 N 次。
4. **效果**：Cypher 一次成功率从 ~70% 提升至 **96%+**，二次重试后接近 100%。

**经验总结**：不同模型分层调用（主链路用便宜快的，关键校验用强模型），既保证准确率又显著降本。

---

### 难点 2：实体对齐召回不准——用户口语化表达 vs 数据库标准命名

**问题表现**：用户问"Java 集合"，数据库里叫"Java 容器与集合框架"；用户问"多线程"，库里叫"并发编程"。单纯向量检索误召回率高，单纯 BM25 又会漏掉语义相近的项。

**解决方案 —— 混合检索 + 异步并发**：

1. **混合打分**：`score = α * vector_score + (1-α) * fulltext_score`，α 默认 0.5，阈值 0.5 过滤低质量召回。
2. **索引规范化**：每个节点类型同时建立 `{label}_vector_index`（768 维 cosine）和 `{label}_fulltext_index`（CJK 分析器）。
3. **并发对齐**：用 `ThreadPoolExecutor(max_workers=10) + asyncio.wrap_future` 把多个实体的对齐查询并发到 Neo4j，避免串行等待。
4. **白名单过滤**：`Teacher`、`Student`、`Price` 三类显式跳过对齐（业务无关 / 隐私）。
5. **效果**：实体对齐准确率从 62% → 91%，对齐阶段平均耗时从 1200ms → 380ms。

---

### 难点 3：流式输出中工具调用的中间 token 污染最终答案

**问题表现**：LangGraph Agent 在执行工具时，工具节点的中间 LLM 输出（如校验结果、抽取结果）也会被 `astream_events` 抛出，直接 yield 给前端会让用户看到一堆中间 JSON。

**解决方案 —— 节点级别的事件过滤**：

```python
async for event in agent.astream_events(..., version="v2"):
    if event["event"] == "on_chat_model_stream" \
        and event["metadata"].get("langgraph_node") == "model":
        yield event["data"]["chunk"].content
```

仅 yield `langgraph_node == "model"` 的事件（即最终回答节点），工具节点的中间输出被丢弃。前端 SSE 流变得干净、纯粹。

---

### 难点 4：多节点部署下的对话上下文连续性

**问题表现**：上线双节点后，`InMemorySaver` 是进程内存级的，用户第一次请求路由到节点 A，第二次路由到节点 B，对话记忆直接丢失。

**解决方案 —— Nginx ip_hash + Session 双保险**：

1. **Nginx 层**：upstream 配置 `ip_hash`，同一客户端 IP 始终路由到固定节点。
2. **应用层**：用 Starlette `SessionMiddleware`（HMAC 签名 Cookie）持久化 `session_id`，作为 LangGraph 的 `thread_id` 实现 checkpointer。
3. **重置机制**：`POST /new-chat` 接口主动清除 session，下次请求生成新 `session_id`。
4. **取舍**：未引入 Redis 共享存储，是因为评估后认为 `ip_hash` 已能覆盖 99% 场景，且引入 Redis 会增加运维复杂度——**架构决策要匹配业务规模**。

---

### 难点 5：流式响应在 Nginx 层被缓冲，token 不实时

**问题表现**：开启 Gzip 和默认 `proxy_buffering` 后，前端要等几秒才能一次性收到一大段文字，体验像非流式。

**解决方案**：

```nginx
location /smart/edu/ {
    proxy_buffering off;        # 关闭缓冲
    proxy_cache off;
    proxy_read_timeout 120s;
    proxy_send_timeout 120s;
    chunked_transfer_encoding on;
    # Gzip 对该 location 关闭，避免压缩缓冲
}
```

**效果**：首字延迟从 ~3s 降至 < 800ms，用户感知"打字机"流畅效果。

---

### 难点 6：UIE 知识点抽取需要 GPU，与 Web 服务部署解耦

**问题表现**：试题文本里要抽取细粒度知识点（如"HashMap 扩容机制"），UIE 模型推理依赖 GPU，但 Web 服务部署在 CPU 节点上。

**解决方案 —— 离线同步 + 在线服务分离**：

1. UIE 推理只在数据同步管道（`sync_cli.py`）中运行，部署在带 GPU 的离线机器。
2. 抽取结果直接写入 Neo4j，Web 服务只读 Neo4j，无需 GPU 依赖。
3. `uie_pytorch/` 目录通过 `sync_cli.py` 在运行时动态 `sys.path.append`，避免污染主项目依赖。

## 六、项目成果与个人收获

### 6.1 项目成果

- **准确率**：知识点关联召回准确率从 62% → **91%**

- **稳定性**：Cypher 一次成功率 **96%+**，双节点部署实现零停机热更新

### 6.2 个人收获

1. 深入掌握 **LangGraph Agent** 工程化落地（`create_agent` / `astream_events` / `checkpointer`）
2. 落地 **RAG + 知识图谱** 融合方案，理解纯向量 RAG 的局限与图查询的价值

5. 完整经历从模型微调（UIE）→ 数据同步 → Agent 编排 → Web 服务 → 多节点部署 的全链路
